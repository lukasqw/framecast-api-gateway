# API Gateway CI/CD Workflows

Este documento descreve os workflows de CI/CD específicos para o API Gateway, separados do fluxo principal da aplicação Go.

## 📋 Visão Geral

O API Gateway possui três workflows principais:

1. **Gateway CI** - Validação contínua em PRs e pushes
2. **Gateway Deploy** - Deploy automático para AWS
3. **Gateway Release** - Criação de releases e tags

## 🔄 Workflows

### 1. Gateway CI (`gateway-ci.yml`)

**Trigger:**

- Pull requests para `develop` ou `main` que modificam `api-gateway/**`
- Push para `develop` que modifica `api-gateway/**`

**Jobs:**

#### validate-terraform

- Valida formatação do Terraform (`terraform fmt`)
- Inicializa e valida configuração Terraform
- Garante que a infraestrutura está correta

#### validate-openapi

- Instala dependências Python
- Executa `build-openapi.py` para gerar spec
- Valida JSON do OpenAPI
- Faz upload do spec como artefato

#### test-lambda

- Testa funções Lambda (main e auth)
- Executa testes unitários com npm
- Valida dependências

#### lint-lambda

- Executa ESLint no código Lambda
- Garante qualidade do código JavaScript/Node.js

#### security-scan

- Audita dependências npm com `npm audit`
- Executa Trivy para scan de segurança
- Identifica vulnerabilidades críticas e altas

### 2. Gateway Deploy (`gateway-deploy.yml`)

**Trigger:**

- Merge de PR de release para `main`
- Dispatch manual com opções:
  - `skip_lambda`: Pular deploy de Lambdas
  - `environment`: production ou staging

**Stages:**

#### Stage 1: Build & Validate

- Extrai versão do branch de release
- Builda OpenAPI spec
- Cria pacotes ZIP das funções Lambda
- Faz upload de artefatos

#### Stage 2: Deploy Infrastructure

- Baixa artefatos (OpenAPI + Lambda packages)
- Configura credenciais AWS
- Executa Terraform plan e apply
- Captura outputs (API endpoint, API ID)

#### Stage 3: Deploy Lambda Functions

- Atualiza código das funções Lambda via AWS CLI
- Usa `aws lambda update-function-code`
- Aguarda propagação das mudanças

#### Stage 4: Integration Tests

- Testa health do API Gateway
- Executa `test-endpoints.sh`
- Executa `test-auth.sh`
- Valida fluxo completo

#### Stage 5: Deployment Summary

- Gera resumo no GitHub Actions Summary
- Inclui links para AWS Console
- Fornece comandos curl para testes

**Secrets Necessários:**

```
AWS_ACCESS_KEY_ID
AWS_SECRET_ACCESS_KEY
AWS_SESSION_TOKEN
JWT_SECRET_KEY
BACKEND_URL
```

### 3. Gateway Release (`gateway-release.yml`)

**Trigger:**

- Push para `develop` que modifica `api-gateway/**`
- Dispatch manual com `release_type` (patch/minor/major)

**Fluxo:**

1. **Verifica PR Aberto**
   - Busca PRs de release existentes (`release/gateway-*`)
   - Se encontrar, atualiza o branch com develop

2. **Cria Nova Release** (se não houver PR aberto)
   - Calcula próxima versão baseada em tags `gateway-v*`
   - Cria branch `release/gateway-vX.Y.Z`
   - Gera changelog filtrado por `api-gateway/`
   - Cria PR draft para `main`

3. **Atualiza PR Existente**
   - Faz merge de develop no branch de release
   - Atualiza changelog com commits recentes
   - Mantém checklist e instruções

**Convenção de Tags:**

- Formato: `gateway-v{major}.{minor}.{patch}`
- Exemplo: `gateway-v1.2.3`
- Separado das tags da aplicação principal

## 🚀 Fluxo de Trabalho

### Desenvolvimento Normal

```bash
# 1. Desenvolver no branch de feature
git checkout -b feature/gateway-new-endpoint
# ... fazer mudanças em api-gateway/ ...
git commit -m "feat(gateway): add new endpoint"
git push origin feature/gateway-new-endpoint

# 2. Criar PR para develop
# CI roda automaticamente (gateway-ci.yml)

# 3. Merge para develop
# Gateway Release workflow cria/atualiza PR de release
```

### Criar Release Manual

```bash
# Via GitHub Actions UI:
# 1. Ir para Actions > Gateway Release
# 2. Click "Run workflow"
# 3. Selecionar release_type (patch/minor/major)
# 4. Confirmar
```

### Deploy para Produção

```bash
# 1. Revisar PR de release criado automaticamente
# 2. Marcar como "Ready for review"
# 3. Aprovar e fazer merge para main
# 4. Gateway Deploy workflow executa automaticamente
```

### Deploy Manual

```bash
# Via GitHub Actions UI:
# 1. Ir para Actions > Gateway Deploy
# 2. Click "Run workflow"
# 3. Configurar:
#    - skip_lambda: false (default)
#    - environment: production (default)
# 4. Confirmar
```

## 📦 Artefatos

### OpenAPI Spec

- **Nome:** `openapi-spec`
- **Arquivo:** `openapi-spec.json`
- **Retenção:** 7 dias (CI) / 1 dia (Deploy)
- **Uso:** Deploy do API Gateway

### Lambda Packages

- **Nome:** `lambda-packages`
- **Arquivos:**
  - `lambda-package.zip` (função principal)
  - `auth-lambda-package.zip` (autorizador)
- **Retenção:** 1 dia
- **Uso:** Deploy das funções Lambda

## 🔐 Segurança

### Secrets Obrigatórios

| Secret                  | Descrição                 | Usado em |
| ----------------------- | ------------------------- | -------- |
| `AWS_ACCESS_KEY_ID`     | Credencial AWS            | Deploy   |
| `AWS_SECRET_ACCESS_KEY` | Credencial AWS            | Deploy   |
| `AWS_SESSION_TOKEN`     | Token de sessão AWS       | Deploy   |
| `JWT_SECRET_KEY`        | Chave secreta JWT         | Deploy   |
| `BACKEND_URL`           | URL do backend            | Deploy   |
| `GITHUB_TOKEN`          | Token GitHub (automático) | Release  |

### Permissões

```yaml
permissions:
  contents: write # Criar tags e releases
  packages: write # Publicar artefatos
  pull-requests: write # Criar e atualizar PRs
  id-token: write # AWS OIDC (futuro)
```

## 🧪 Testes

### Testes Locais

```bash
# Validar Terraform
cd api-gateway
terraform fmt -check -recursive
terraform init -backend=false
terraform validate

# Validar OpenAPI
python scripts/build-openapi.py
python -m json.tool openapi-spec.json

# Testar Lambda
cd lambda
npm ci
npm test

cd auth
npm ci
npm test

# Lint
npm run lint
```

### Testes de Integração

```bash
# Após deploy
cd api-gateway
./scripts/test-endpoints.sh
./scripts/test-auth.sh
```

## 📊 Monitoramento

### GitHub Actions Summary

Cada deploy gera um resumo com:

- Versão deployada
- Ambiente (production/staging)
- API Gateway URL e ID
- Status de cada stage
- Links para AWS Console
- Comandos curl para testes

### Logs

```bash
# Ver logs do workflow
gh run list --workflow=gateway-deploy.yml
gh run view <run-id> --log

# Ver logs do Terraform
# Disponível nos logs do job "deploy-infrastructure"

# Ver logs das Lambdas
aws logs tail /aws/lambda/oficina-tech-api-lambda --follow
aws logs tail /aws/lambda/oficina-tech-auth-lambda --follow
```

## 🔧 Troubleshooting

### Terraform Apply Falha

```bash
# Verificar estado
cd api-gateway
terraform init
terraform plan

# Ver outputs
terraform output

# Forçar unlock (se necessário)
terraform force-unlock <lock-id>
```

### Lambda Update Falha

```bash
# Verificar função existe
aws lambda get-function --function-name oficina-tech-api-lambda

# Ver configuração
aws lambda get-function-configuration --function-name oficina-tech-api-lambda

# Testar manualmente
aws lambda invoke --function-name oficina-tech-api-lambda output.json
```

### OpenAPI Build Falha

```bash
# Validar arquivos JSON
cd api-gateway/openapi
for file in paths/*.json; do
  echo "Validating $file"
  python -m json.tool "$file" > /dev/null
done

# Rebuild
python scripts/build-openapi.py
```

## 🔄 Migração de Repositório

Quando separar em 2 repos:

### Repo 1: Aplicação Go (oficina-tech-api)

Manter:

- `.github/workflows/ci.yml`
- `.github/workflows/deploy.yml`
- `.github/workflows/release.yml`

### Repo 2: API Gateway (oficina-tech-gateway)

Mover:

- `api-gateway/` → raiz do novo repo
- `.github/workflows/gateway-*.yml` → `.github/workflows/`
- Renomear workflows:
  - `gateway-ci.yml` → `ci.yml`
  - `gateway-deploy.yml` → `deploy.yml`
  - `gateway-release.yml` → `release.yml`

Ajustar paths nos workflows:

```yaml
# Remover prefixo api-gateway/
- 'api-gateway/**'  →  '**'

# Ajustar working directory
WORKING_DIR: api-gateway  →  WORKING_DIR: .
```
