# GitHub Workflows

Este repositório contém três workflows principais para CI/CD do API Gateway.

## Workflows

### 1. CI (Integração Contínua)

**Arquivo:** `.github/workflows/ci.yml`

**Triggers:**

- Pull requests para `develop` ou `main`
- Push para `develop`

**Jobs:**

- `validate-terraform`: Valida formatação e configuração Terraform
- `validate-openapi`: Valida e constrói a especificação OpenAPI
- `test-lambda`: Executa testes das funções Lambda
- `lint-lambda`: Verifica qualidade do código Lambda
- `security-scan`: Escaneia vulnerabilidades de segurança

### 2. Deploy (Implantação)

**Arquivo:** `.github/workflows/deploy.yml`

**Triggers:**

- Merge de PR de release para `main`
- Execução manual via `workflow_dispatch`

**Jobs:**

- `build-validate`: Constrói e valida artefatos
- `deploy-infrastructure`: Implanta infraestrutura Terraform
- `deploy-lambda`: Atualiza funções Lambda
- `integration-tests`: Executa testes de integração
- `deployment-summary`: Gera resumo da implantação

### 3. Release (Criação de Release)

**Arquivo:** `.github/workflows/release.yml`

**Triggers:**

- Push para `develop`
- Execução manual via `workflow_dispatch`

**Funcionalidade:**

- Cria automaticamente branches de release
- Gera changelog
- Cria Pull Request para `main`
- Atualiza PRs existentes com novas mudanças

## Configuração de Secrets

Os workflows requerem os seguintes secrets configurados no GitHub:

### Obrigatórios para Deploy:

- `AWS_ACCESS_KEY_ID`: Chave de acesso AWS
- `AWS_SECRET_ACCESS_KEY`: Chave secreta AWS
- `AWS_SESSION_TOKEN`: Token de sessão AWS (se usando credenciais temporárias)
- `JWT_SECRET_KEY`: Chave secreta para JWT
- `BACKEND_URL`: URL do backend da aplicação

### Opcional para Release:

- `PAT_TOKEN`: Personal Access Token com escopo `repo` (recomendado)

## Configurando Personal Access Token (PAT)

O workflow de release pode falhar ao criar PRs devido a limitações de permissão do `GITHUB_TOKEN` padrão. Para resolver isso:

### 1. Criar um PAT:

1. Acesse: GitHub → Settings → Developer settings → Personal access tokens → Tokens (classic)
2. Clique em "Generate new token (classic)"
3. Dê um nome descritivo: "Gateway Release Workflow"
4. Selecione os escopos:
   - `repo` (acesso completo a repositórios)
5. Clique em "Generate token"
6. Copie o token (você não poderá vê-lo novamente!)

### 2. Adicionar como Secret:

1. Vá para o repositório → Settings → Secrets and variables → Actions
2. Clique em "New repository secret"
3. Nome: `PAT_TOKEN`
4. Valor: Cole o token copiado
5. Clique em "Add secret"

### 3. Alternativa sem PAT:

Se você não configurar o PAT, o workflow irá falhar ao criar o PR automaticamente, mas:

- A branch de release será criada
- Você pode criar o PR manualmente
- O workflow fornecerá instruções claras no log

## Estrutura de Branches

```
main (produção)
  ↑
  └── release/gateway-vX.Y.Z (release)
        ↑
        └── develop (desenvolvimento)
              ↑
              └── feature/* (features)
```

## Fluxo de Trabalho

1. **Desenvolvimento:**
   - Crie branches `feature/*` a partir de `develop`
   - Abra PR para `develop`
   - CI executa automaticamente

2. **Release:**
   - Push para `develop` dispara workflow de release
   - Cria branch `release/gateway-vX.Y.Z`
   - Cria PR para `main`
   - Revise e aprove o PR

3. **Deploy:**
   - Merge do PR de release para `main`
   - Deploy automático para AWS
   - Testes de integração executados

## Versionamento

O projeto usa versionamento semântico com prefixo `gateway-v`:

- `gateway-v1.0.0`: Release de produção
- Formato: `gateway-vMAJOR.MINOR.PATCH`

Tipos de release (via `workflow_dispatch`):

- `major`: Mudanças incompatíveis (1.0.0 → 2.0.0)
- `minor`: Novas funcionalidades compatíveis (1.0.0 → 1.1.0)
- `patch`: Correções de bugs (1.0.0 → 1.0.1)

## Troubleshooting

### Erro: "GitHub Actions is not permitted to create or approve pull requests"

**Solução:** Configure o `PAT_TOKEN` conforme instruções acima.

### Erro: "terraform fmt -check failed"

**Solução:** Execute localmente `terraform fmt -recursive` e commit as mudanças.

### Erro: "npm ci failed - no package-lock.json"

**Solução:** Os arquivos `package-lock.json` devem estar commitados no repositório.

### Erro: "terraform validate failed - file not found"

**Solução:** O workflow agora constrói os arquivos necessários automaticamente antes da validação.

## Monitoramento

Após o deploy, você pode monitorar:

- **API Gateway:** AWS Console → API Gateway
- **Lambda Functions:** AWS Console → Lambda
- **Logs:** CloudWatch Logs
- **Métricas:** CloudWatch Metrics

Comandos úteis:

```bash
# Ver logs do API Gateway
make logs-api

# Ver logs do Lambda Authorizer
make logs-lambda

# Ver logs do Lambda Auth
make logs-lambda-auth

# Ver outputs do Terraform
make output
```
